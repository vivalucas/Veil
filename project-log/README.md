# Veil Project Log 规范

本目录是 Veil 的内部开发知识库。与主 `README.md` 面向用户不同，`project-log/` 面向维护者和 AI 助手，用来保存项目背景、当前状态、设计决策、部署说明和开发日志。

当前 GitHub 仓库为 private，`project-log/` 可以随代码提交，用于同步内部上下文。准备把仓库改为 public 前，必须先把 `project-log/` 加入 `.gitignore`，并从公开 Git 历史中清理掉此前提交过的 `project-log/` 记录。

## 快速查找

新成员 / 新对话按以下优先级阅读：

1. `05-current-status.md` — 当前进展、待办、阻塞和交接信息。

## 每次开工前必读 / 必做流程

1. 读取 `05-current-status.md`，确认当前阶段、待办、阻塞和下一步。
2. 读取与任务相关的设计文档，例如功能改动读 `01`，自动化改动读 `03`，架构改动读 `04` 和 `10`。
3. 如果当前设计原因不明，或准备推翻已有产品/交互/架构取舍，先查 `12-design-decisions.md`。
4. 改代码前说明计划、影响范围和可能需要同步更新的 project-log 文件。
5. 涉及较大改动时，先在 `10-planning-log.md` 记录方案并确认，再动代码。
6. 改完后更新 `05-current-status.md` 和 `06-dev-log.md`。
7. 涉及功能、数据、自动化接口、架构、部署、环境变量或外部服务时，同步更新对应编号文档。
8. 如果本轮形成了长期设计取舍，或放弃了未来可能再次被提出的方案，同步更新 `12-design-decisions.md`。
9. 无法验证、发现文档与代码不一致、或遇到不确定决策时，必须明确写出。
10. **清理构建产物**：开发机与日常使用机共用，完成测试或构建后，及时删除 DerivedData、build 目录、测试生成的 .app 等产物，避免 Spotlight 索引后干扰日常搜索。

## 文档哲学：只增不删，改动留痕

- 只增不删：过时内容划掉并注明原因，不直接删除。
- 记录「改动 + 原因」：不只写结果，要写为什么这么做。
- 保留被否定的方案：「为什么不用 X」和「为什么用 Y」同样有价值。
- 不适用也要标注：某节暂时不用时，写明「不适用」和原因。

修改已有内容的方式：

```markdown
<!-- 旧方案（废弃于 2026-05-19，原因：示例） -->
~~旧方案内容~~

<!-- 新方案 -->
新方案内容，详见 10-planning-log.md。
```

## 开发日志格式

每轮开发在 `06-dev-log.md` 中追加一条记录：

```markdown
## YYYY-MM-DD（主题描述）

**触发原因**：

**修改内容**：
1. 文件路径 — 改了什么、为什么改

**遇到的问题**：
- 问题描述

**解决方式**：
- 如何解决的

**验证方式**：
- 运行了什么测试、命令，或做了什么手动检查

**验证结果**：
- 通过 / 未通过 / 未运行及原因
```

## Git Commit 格式

```text
<type>: <简短描述>
```

| 类型 | 用途 |
|------|------|
| `feat` | 新增功能 |
| `fix` | Bug 修复 |
| `refactor` | 重构（不改变外部行为） |
| `docs` | 文档更新 |
| `style` | 样式/格式调整（不影响逻辑） |
| `chore` | 构建、依赖、配置等杂项 |

每次提交对应一个清晰的意图。如果一次改动包含多个独立目的，拆成多个提交。

## 安全与敏感信息

- 不要把真实 token、password、private key、cookie、连接串密码写入 project-log。
- 示例值必须使用假值，例如 `sk-xxx`、`example-password`。
- 如果发现敏感信息曾被提交或写入文档，在 `06-dev-log.md` 记录处理方式，并立即轮换对应密钥。

## 主 README 与 project-log 的分工

| 内容 | 主 README | project-log |
|------|-----------|-------------|
| 项目简介 / 功能介绍 | 写 | 不写 |
| 安装指南 | 写 | 不写 |
| 技术栈概览 | 简要写 | 详细写 |
| 数据存储设计 | 不写 | `02` |
| URI scheme 详细定义 | 链接 | `03` 和 `docs/URI_SCHEMES.md` |
| 架构决策 | 不写 | `04` + `10` + `12` |
| 开发日志 | 不写 | `06` |
| 发布详细步骤 | 简要写 | `07` + `08` |
| 当前进度 | 不写 | `05` |

原则：主 README 给外部用户看，project-log 给自己人看。
